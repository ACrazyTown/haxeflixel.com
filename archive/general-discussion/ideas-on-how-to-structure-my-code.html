Ideas on how to structure my code?

Uhfgood, 3 months 2 days ago

So I made this game - http://www.ludumdare.com/compo/ludum-dare-26/?action=preview&uid=53 for Ludum Dare 26 (the one before the last one, as I didn't participate in 27) -- And am in the process of building a framework to port this over from Unity to haxe/flixel.

So with some great help of some friends in the irc channels #haxe #openfl #haxeflixel on the freenode network I managed to create a json loader.  Now is the time I'm supposed to actually start making a game. 

So the game is (and future games are going to be) a choose-your-own-adenture-style game.  Where each "page" is called a Scene.  Eventually I'm going to blow this into into a full blown graphic adventure game engine, but right now I'm just going to have pages, with graphics and text and some basic buttons. 

So a scene contains scenery objects, which are essentially images with a position and size, which themselves, also contain animations.  Animations right now are just going to be a collection of individual frames that I'm going to use for the buttons (visual feedback as to whether you pressed them or not).  Okay so it goes like this Scene->scenery_objects->animation

Sorry for the long-winded explanation but to get to my point I'm trying to decide if I should just do one FlxState for the whole game, where my scenes are contained.  Or if each scene should actually be extended from FlxState (meaning each scene is it's own state).

I could see some advantages for both.  For instance if it's just one state then i have access to every scene and scenery object, but it also means manually changing out what backgrounds and stuff get added.  So it will be a lot of adding and removing when switching scenes.  On the other hand individual states I can just use Flixels state management to switch scenes so I could just add everything in one go, and in fact have each scene add and remove it's own objects before switching.

Really it seems a bit more ideal to have each scene be a state, but that makes the framework tied to haxeflixel more than the other way.  Makes it a bit trickier if I want to move frameworks to say, haxepunk (if there was ever a reason to), or to plain openfl (if I ever learn how to properly use openfl plain or just want more control).

So I'm looking for suggestions on what to do.  Any ideas?

I should probably add another advantage to only one state is that I can use the same code for the logic on all states.  (While not a lot of realtime logic will be taking place at this time, at least for clicking buttons, I can use one set of code rather than having to rewrite it for each state).

Also if I make each scenery object a flixel sprite, then that ties it even more together, but I might kind of want that.

Either way could work, the main advantage of inheriting FlxState is easy access to add() and remove() calls from FlxGroup. The most important thing to keep your code modular, is to create your own base layer classes. Like: MyCustomState extends FlxState, or MyCustomSprite extends FlxSprite.

That way you're not tied down to anything, since you can always build on top of your base classes and extend them however you want. Good programming practices aren't developed in a short time, you need to do a lot of projects, keep practicing and don't give up.

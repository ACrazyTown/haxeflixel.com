Well as far as I can tell its mostly about a name change, focus on a consistent api and getting better tests written. It certainly doesn't loose anything NME had before the change. What I dont know much about is the details and impact of the some of the refractor of the tools mentioned.

The name will certainly receive the same superficial criticism flash does unfortunately, however people who understand why the flash api and actionscript like languages are so widely being "saved" and used in things like StageXL,  Scaleform, Loom etc. NME has always been about the flash api so I think its an honest and strong move forward.

I think the long term challenge for openfl will be getting developers to understand its not about the 'flash runtime' its about the language, workflow and toolset for future runtimes without Adobe. I agree that focus on webgl, ecmascripten and use of Haxe makes openfl a great answer to people who were counting on AS NEXT that Adobe killed.

https://plus.google.com/100343108333442407381/posts/CZaEsN8cLv9

Bullet points from today's #NME announcements at #wwx  
(as far as I gathered, so correct me!)

* Rebrand from NME to #OpenFL (Open Flash Library) to give a proper unique name and aid web searches and community efforts
* New focus on codebase flexibility and extensibility
* New focus on testable code, with all features undergoing unit testing across targets
* Haxe3 only
* Aims to offer stable, reliable OpenGL contexts on native targets and HTML5 through WebGL
* WebGL API used for OGL on native targets (this is a good thing IMHO)
* No specific aid to use Stage3D target (though with H3D and HXSL I personally wonder what more aid we'd ever need for that)
* Officially backed by the Haxe Foundation
* Codebase restructured to omit nme package; Common root package for all targets now is "flash", while library additions are under "openfl"
* openfl.org is the website, but it's currently under construction (Where's the animated construction worker gif Joshua?!)

Beeblerox
6 months 1 week ago

HaxeFlixel works with OpenFl already. Main conversion you'll have to make is to fix imports (replace nme.* with flash.* in most cases). Engine have same features and keep evolving, son don't afraid of this change.

OpenFl codebase has been splitted in several functional parts (openfl - base lib, openfl-tools - command line tools, openfl-native - implementation for native targets and openfl-html5 - html5 target). It allows to update them independently and make it faster, so i believe it is a good change.

OpenFl isn't just rebranding of good old NME, but it's a redefined architecture, and i hope it will make this lib faster and easier to understand and work with for newcomers (especially from ActionScript background)

No, you shouldn't really have any trouble with it. If you use FlashDevelop though, you should make sure to use the latest build (4.4.3) which supports compiling via openfl. You just need to rename the .nmml to an .xml file which tells FD you're using openfl - don't forget to change the build file in the project properties as well. Or get the latest HaxeFlixel version from GitHub, which already has that change.

Recent Flashpunk convert -- trying to transfer my knowledge

jacobalbano, 2 months 1 week ago

Hey there :)

I've been using Flashpunk for a few years, but I'm transitioning to HaxeFlixel in order to make use of Haxe's fantastic cross-platform support. So far I like the framework a lot (and it's given me way less trouble than Haxepunk), but there are some things that I feel are missing, or maybe I'm just looking in the wrong place.

  • In Flashpunk, I'm used to designing my game objects as autonomous entities that are loaded from map files and never interact explicitly. The Flixel examples seem to all involve a FlxState that includes a lot of logic and hard references to objects. Is it possible to handle collision and interactions between objects from within each object?
  • Flashpunk has a bunch of helper functions for collision like moveBy() and moveTowards(), which, when given a delta or destination, will move an object as far as it can go towards its target before it collides and stops just short of the collision. Are there similar features in Flixel? I haven't been able to find them.
  • In Flashpunk I can override the moveCollide() function and decide which objects are collidable based on their type; for example, I can have it so that pawns of the same color can move through each other, but different colors trigger a collision. Can I do this in Flixel as well, or is it just a matter of manually grouping objects and running the collisions manually?

 

I've been able to intuit a lot so far, but I think my main problem here is that Flixel's approach to a lot of things is fundamentally different than Flashpunk's. I'd appreciate any help you folks are able to give to get me on the right track.

This was also posted over on the Flixel forums, but I thought it might be more appropriate here since I'm using HaxeFlixel specifically. Hope there isn't a problem with that.

  • Sure, you can check for collisions within individual FlxObjects as well. (Either via the overlaps() function or just the "traditional" FlxG.collide(). You'll still be required to have a FlxState, and generally, when you're doing a lot of collision checks, it's faster to do it by colliding FlxGroups rather than colliding individual objects.
  • I don't think there's anything exactly like that... You might find the functions in FlxVelocity helpful, but they don't take collisions into account afaik. You could possibly combine some of them with FlxG.collide().
  • FlxG.collide() is really only FlxG.overlap() with the seperation functions of FlxObject being called afterwards (FlxObject.seperateX() and FlxObject.seperateY()). So you can accomplish this by calling overlap(), then doing your checks in the callback function and seperate the objects depending on their color or whatever.

  • In Flashpunk I can override the moveCollide() function and decide which objects are collidable based on their type; for example, I can have it so that pawns of the same color can move through each other, but different colors trigger a collision. Can I do this in Flixel as well, or is it just a matter of manually grouping objects and running the collisions manually?

A relatively simple way to do this is to have separate FlxGroups (that you never add to FlxState) that are used only for collision checks. It's a bit more work upfront to create the groups and make sure you destroy them properly (make sure you clean up your references manually since you're not adding those groups to FlxState, which cleans up any added objects automatically), but once you have your setup working, FlxGroups simplify a lot of things for you.

Quote:
In Flashpunk, I'm used to designing my game objects as autonomous entities that are loaded from map files and never interact explicitly. The Flixel examples seem to all involve a FlxState that includes a lot of logic and hard references to objects. Is it possible to handle collision and interactions between objects from within each object?

I believe it is possible to handle collision within each object. FlxObject has an isToucing() function that you can use to determine what side/sides a FlxObject (and inheriting classes) is being touched on. You can also reference FlxG's static functions overlap/collide, but you will need a sprite or group to compare against in this case.

Quote:
Flashpunk has a bunch of helper functions for collision like moveBy() and moveTowards(), which, when given a delta or destination, will move an object as far as it can go towards its target before it collides and stops just short of the collision. Are there similar features in Flixel? I haven't been able to find them.

I don't think there is a built in way to solve these problems specifically. There are some functions in Util.FlxMath that may be useful and there is a pathfinding demo that could help as well. You will likely have to write your own code to handle these situations.

Quote:
In Flashpunk I can override the moveCollide() function and decide which objects are collidable based on their type; for example, I can have it so that pawns of the same color can move through each other, but different colors trigger a collision. Can I do this in Flixel as well, or is it just a matter of manually grouping objects and running the collisions manually?

I think it would be best to group objects appropriately and use collide(). Basic collision is quite simple as long as you group your objects nicely. After that you just have to call FlxG.collide(object/group,object/group) in update and you're golden.

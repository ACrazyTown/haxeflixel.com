HaxeFlixel UI plugin development?

larsiusprime, 10 months 2 weeks ago

Hey guys, I had a random question...

 

I feel like 90% of the work in my games is user interface design, and the only real thing that I feel like I'm missing from the old, slow-as-butt vanilla flash is the great UI stuff.  Most flixel games are pretty action-oriented so the lack of a lot of nice user interface widgets isn't always a big deal. For me though, it's a gap in the framework and I figure this is a wheel we could all do without reinventing.

 

So, when I start getting underway with Defender's Quest 2 I was thinking it might be smart to split off the user interface development as a new FlxPowerTools plugin?

 

Has anything like this been done before beyond basic stuff like buttons?

 

Here's what I'd like to cover:

 

-xml-based layout system that lets you generate user-interfaces without having to hard-code and position everything

-radio buttons

-checkboxes

-input text fields

-scroll buttons / scroll bar

-(obviously everything should be skinnable)

-tooltips

-tab navigation

 

I've implemented a bunch of this stuff in an okay-but-not-great way in Defender's Quest 1, but I would like to improve the quality. I don't have a lot of experience with working on collaborative projects and making code that's up to those kind of standards, so I'd love to swap notes and learn how to contribute and give back to a cool project like this and have everyone benefit from it.

 

Would this be valuable to anyone?

Sorry I've been a little quiet on this front for a while, I had to do some maintenance on our old as3 engine which distracted me from haxe work.

I'm currently working on adding "anchoring" to haxe-flx-ui. This will allow the system to deal gracefully with unpredictable screen sizes, and will be essential for both PC and mobile development.

One issue in our own game, Defender's Quest, is that the UI needs to change in certain layouts - the battle map will create more dead space on either the top of the side depending on whether it's a 3:4 or 16:9 screen size, etc.

Here's a mockup at 1024x768 resolution (ie, iPad 1 screen size):

And here's a mockup at 1920x1080 (ie, HD "1080p" resolution):

In the first layout, there's more dead space on the top, so we lay out the UI that way, and put the wave bar on the side. On the bottom mockup, there's more space on the side, so we put the UI there and the wave bar up top.

So, in the next update to haxe-flx-ui, you'll be able to specify different layouts, and position elements with auto-resizing and relative anchors. Ultimately, you'll create your layouts, and then set your resolution to anything, and the engine will try to find the optimum layout based on the current resolution.

Okay, so here's what's new:

https://github.com/haxeflixel/flixel-ui

1. Default assets
2. Loading widgets with raw code (no XML)
3. Firetongue (localization) integration
4. Documentation!

All of the information is on the README documentation on the flixel-ui repository. Here's a brief summary of the above:

--If you load a widget without providing graphic assets or style definitions, it will use simple default ones that come with the library.
--You can load widgets with XML, though it might be a little verbose. Examples are included in the documentation.
--You can hook this up to my localization library, Firetongue, to get both translated text, as well as per-locale adjustments to UI elements (so you can make buttons or text windows bigger in German, for instance)
--Everything is documented in the main readme (I think!)

I've also refactored things a bit for efficiency and renamed all the widgets to "FlxUIWhatever" to avoid confusion (I dropped the previous FlxWhateverX convention).

This should accomplish pretty much all the stuff I've been wanting to add for a long time. Now just big stuff (UI editor), bug fixes, and slowly adding more widgets is left.

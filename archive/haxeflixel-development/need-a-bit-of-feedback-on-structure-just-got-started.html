It's a simple project now, so you don't need to think too much about code organisation :) Focus on gameplay and ideally treat it as a prototype, that's going to be trashed some day. Then start from the beginning.

One thing is that you will have a playable prototype faster. The other, as important, is that you will waste less time. You will know by then what pieces of codes are too complicated, hard to read or cause problems. It will be obvious what type of code organisation you want for this project.

The book Code Complete, 2nd ed. is full of perfect examples on good design of software. It's brilliantly written. You can go straight the topics you're interested in, then read everything else for broader knowledge.

(I think it's better to read it even quickly, than to follow simple rules like "split into as much classes as you can". It's mostly true, but you know :P such simple tricks are not knowledge.)

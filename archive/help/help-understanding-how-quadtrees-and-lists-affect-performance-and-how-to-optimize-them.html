Help understanding how Quadtrees and Lists affect performance and how to optimize them

Monomer, 1 month 3 weeks ago

Howdy folks; I've been working on a hobby game that's vaguely similar to Super Puzzle Platformer deluxe  (http://store.steampowered.com/app/238530/) in the sense that it's a falling-block puzzle game where you control a dude in the puzzle field and move the blocks with your character.

I haven't gotten too far, though, and I started seeing performance concerns. My grid in the game is 10x14, so it could theoretically have up to 140 blocks on screen at one time, and they are making use of overlap checks to make sure they land and stop where they're supposed to when falling; at first, I was simply checking for overlap between all members of the block group (a FlxTypedGroup from which I recycle up to 140 Block type objects) against themselves and the "walls" of the play area, which is when I noticed performance dropping as I added more sprites.

Some google searching told me that the Quadtrees and Lists counts in the debugger are related to collision/overlap checks, and as I added more blocks they did get pretty high (40 blocks = 200 lists and 85 quadtrees, approx.) I tried to optimize this in a variety of ways, and currently have it implemented so that the overlap calls are only called between a group that only contains the currently moving blocks, vs. a group that contains exactly 10 invisible colliders which I re-size dynamically to match the number of blocks in any column at a given time. I was hoping this would cut down on the inflation of Quadtrees and Lists because instead of calling overlap against an ever increasing number of blocks, it only does it vs. the 10 colliders and any moving blocks (usually between 1 and 5). Yet, when testing that, the Quadtrees and Lists still went up as I added more blocks to the scene, even when none were moving.

So my question is, what functions cause the Quadtree and List count to go up? Am I probably still somehow using the bounding boxes of every block even though I'm not calling collide or overlap against all of them? How does this tend to affect performance (am I correct to assume that the Quadtree and List counts are a likely contributer to my slow framerate)?

Any help would be appreciated; I'm not looking for a magic answer for my particular project, but just a better understanding of how to optimize HaxeFlixel when dealing with a lot of colliding objects. Thanks!

NOTES: I'm using the current dev branch of HaxeFlixel and the most recent OpenFL, and this behavior was seen in both Flash and Neko (Mac and PC) targets.

It's likely there's an issue with how you're adding the blocks and wall zones and how you perform collision checks. Could you provide some sample code of how you're structuring the groups that are used for collision checks?

Unfortunately I don't know enough about the collision system to say what could cause so many quad trees to be created. If you run in cpp mode, you can profile your application to find out exactly which function calls take the longest to execute, that could give you some hints on where to focus your optimization efforts. Look in the Built-in Profiler section of the article.
